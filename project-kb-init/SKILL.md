# 项目知识库初始化 Skill（project-kb-init）

## 目标

在项目根目录下创建一套可维护的 `knowledge/` 知识库骨架，支持：

- 按项目类型生成领域目录（例如 Web 项目的前端/后端）
- 每一层目录都带 `data_structure.md` 作为分层索引
- 为已阅读的关键代码文件生成说明性 `.md` 文档，方便后续检索
- 功能模块较大时可继续拆子目录，但每个新目录都必须有 `data_structure.md`

## 严格触发条件

仅当同时满足以下条件时调用本 Skill：

1. 用户明确表达“创建项目知识库”或同义意图（如“初始化 knowledge”“搭建项目知识库结构”）。
2. 项目根目录当前**不存在** `knowledge/` 文件夹。

若 `knowledge/` 已存在：

- 不执行本 Skill 的创建流程。
- 明确告知用户当前项目已有知识库，建议改用维护/补充类流程。

## 执行原则

- 使用绝对资源路径（如 `res://...`）仅适用于 Godot 资源；本 Skill 的知识库目录仍按项目真实文件系统路径创建。
- 不做临时拼凑：目录设计应可长期维护，命名稳定清晰。
- 先理解后落盘：先识别项目类型与核心模块，再创建目录与说明文档。
- 最小惊讶原则：任何会新增“功能模块目录”的动作，都先询问用户确认。
- 大型项目谨慎原则：当项目很大或用户仅指定某一模块时，先缩小范围再处理，避免全仓扫描和无界代码阅读。
- 模块定向创建原则：若用户要求“只创建某个模块知识库”，应先获取模块入口文件夹；若没有入口，仅允许依据目录名/文件名推断候选范围，禁止直接读取代码内容。
- 零越界原则：在“范围未确认”状态下，禁止创建任何模块知识库目录（包括临时目录）与功能说明文档。
- 模式优先级原则：初始化模式（轻量/全量）的规则优先于默认流程规则；发生冲突时以已选模式为准。
- 轻量证据优先原则：轻量模式下仅允许依据项目现有目录名与文件名推理知识库目录，禁止凭经验假设项目结构或直接套用模板。

## 标准流程

### 0) 初始化模式选择（必须）

在创建前先让用户二选一，并明确成本与边界：

- **轻量模式（推荐）**
  - 仅创建知识库框架：目录 + 各层 `data_structure.md`
  - 不读取代码、不生成功能说明文档
  - 基于项目现有目录名/文件名推理目录结构
  - 适合先搭骨架，后续按需逐步构建知识库
- **全量模式**
  - 创建目录与索引，并在确认范围内读取代码生成说明文档
  - 一次性构建较完整的项目知识库
  - 可能消耗大量 token（项目越大越明显）

询问模板（可直接使用）：

`请选择初始化模式：轻量模式（仅框架，后续按需逐步构建） / 全量模式（完整构建，可能消耗大量 token）。`

触发后首句回复模板（可直接使用）：

`我可以为你初始化项目知识库。开始前请先选择模式：轻量模式（仅构建框架，后续按需逐步补全）或全量模式（完整构建，可能消耗较多 token）。`

默认建议：当用户未明确偏好时，优先推荐轻量模式。

### 1) 前置检查

1. 识别项目根目录（优先用户指定路径，否则当前工作目录）。
2. 检查 `<project_root>/knowledge` 是否存在。
3. 若存在，停止创建并反馈原因。

### 2) 识别项目类型与建议目录

根据已选模式执行：

- **全量模式**：可根据项目特征文件判断主类型，并给出初始目录建议。
- **轻量模式**：禁止直接套用类型模板；必须先基于项目现有目录名/文件名给出候选目录，再请用户确认。

全量模式下，可参考以下项目特征文件判断主类型并给出初始目录建议：

- **Web 项目**（如 `package.json`、`vite.config.*`、`next.config.*`、`pom.xml` + web 模块）
  - 建议一级目录：
    - `knowledge/01_product/`
    - `knowledge/02_frontend/`
    - `knowledge/03_backend/`
    - `knowledge/04_database/`
    - `knowledge/05_deployment/`
- **Godot 项目**（`project.godot`）
  - 建议一级目录：
    - `knowledge/01_gameplay/`
    - `knowledge/02_systems/`
    - `knowledge/03_scenes_assets/`
    - `knowledge/04_data_config/`
    - `knowledge/05_build_release/`
- **大型游戏项目（功能模块制）**
  - 建议一级目录：
    - `knowledge/01_3c/`
    - `knowledge/02_ai/`
    - `knowledge/03_network/`
    - `knowledge/04_engine/`
    - `knowledge/05_ui/`
    - `knowledge/06_audio/`
    - `knowledge/07_tools_pipeline/`
    - `knowledge/08_liveops_ops/`
  - 说明：可按项目实际裁剪；不需要的目录不强制创建。
- **通用后端/服务项目**
  - 建议一级目录：
    - `knowledge/01_domain/`
    - `knowledge/02_services/`
    - `knowledge/03_data/`
    - `knowledge/04_interfaces/`
    - `knowledge/05_ops/`

若无法准确识别，先给出“通用模板 + 识别依据”，并请用户确认。

轻量模式下，候选目录建议输出应至少包含：

- 候选目录名
- 证据来源（目录名/文件名）
- 置信度（高/中/低）
- 是否待用户确认

### 3) 选择创建范围（全项目 / 单模块）

根据用户意图进入两种模式：

- **全项目模式**：按项目模板创建一级领域目录。
- **单模块模式**（如“只建 AI 模块知识库”）：
  1. 先请求用户提供该模块的入口文件夹路径。
  2. 若用户未提供入口文件夹：
     - 先基于目录名与文件名推断候选范围（例如 `ai/`, `npc/`, `behavior_tree`, `net`, `replication`）。
     - 输出候选清单并请用户确认其一（建议 3-7 个候选，附置信度与判定依据）。
     - 在用户确认前，**禁止读取代码内容**。
     - 在用户确认前，**禁止创建模块目录或说明文档**。
  3. 范围确认后，仅在该范围内创建模块知识库目录与索引。

单模块模式建议输出格式：

```md
## 模块范围确认
- 目标模块：AI
- 当前状态：待确认 / 已确认

## 候选入口目录（按置信度排序）
1. `<path-a>`（高）- 依据：目录名/文件名
2. `<path-b>`（中）- 依据：目录名/文件名

## 下一步
- 请选择一个入口目录；确认前我不会读取代码或创建模块知识库目录。
```

### 4) 创建分层索引树（必须）

创建目录时，遵循以下硬性规则：

- `knowledge/` 根目录必须有 `data_structure.md`
- 每个一级领域目录必须有 `data_structure.md`
- 任何新建的二级/三级目录也必须有 `data_structure.md`

根目录 `data_structure.md` 最少包含：

- 知识库目标与使用方式
- 一级目录清单与用途
- 维护约定（新增目录必须补索引、文档命名规范）

子目录 `data_structure.md` 最少包含：

- 当前目录职责边界
- 子目录/文件说明
- 建议沉淀的文档类型

模式化约束：

- **轻量模式**：仅创建“已确认候选目录”的框架与 `data_structure.md`，不创建功能说明文档。
- **全量模式**：创建索引树后可继续执行代码阅读与说明文档生成。

### 5) 阅读代码并生成功能说明文档（仅全量模式）

仅在用户选择全量模式且范围已经确认后执行（全项目模式下可按优先模块逐步执行，单模块模式下仅限已确认范围）：

1. 读取代表性代码文件（控制器/服务/场景脚本/核心模块入口等）。
2. 在对应知识库目录创建说明文档（`.md`），建议命名：
   - `<module_name>_overview.md`
   - `<module_name>_flow.md`
   - `<module_name>_api_or_interface.md`（若适用）
3. 每份说明文档至少包含：
   - 该代码文件/模块的职责
   - 关键流程（输入 -> 处理 -> 输出）
   - 依赖关系（调用谁、被谁调用）
   - 关键数据结构或配置点

注意：除 `data_structure.md` 之外，其余 `.md` 是具体知识内容，不能只写占位标题。

额外约束：

- 若仍处于“待确认范围”状态，本步骤不得执行。
- 大型项目优先“索引先行”：先补齐目录级 `data_structure.md`，再逐步补代码说明文档。
- 若用户选择轻量模式，本步骤必须跳过，并在输出中给出后续“逐步构建建议清单”。

### 6) 大模块拆分时的人机确认（轻量模式必须）

当某功能模块较大，需要创建新的子模块目录时：

轻量模式下：

1. 先给出建议（要创建哪些子目录、原因是什么）。
2. 明确询问用户是否创建。
3. 用户同意后再创建目录及对应 `data_structure.md`。

全量模式下：

- 在用户已选择全量模式且模块范围已确认后，新增规划内子模块目录可直接创建，无需逐一二次询问。

优先级规则：

- 轻量模式下，即使用户在同一条消息里说“你可以直接创建”，只要属于“新增功能子模块目录”（如 `perception/planning/behavior`），仍需先给一轮确认提示，再执行创建。
- 全量模式下，范围确认后不再要求对每个新增功能子模块目录单独确认。

询问模板（可直接使用）：

`该模块较大，我建议新增以下子目录：A / B / C，以便拆分维护。是否现在创建？`

## 文档模板

### A. `data_structure.md` 模板

```md
# 目录索引

## 目录定位
- 本目录负责：

## 子目录
- `sub_dir_a/`：
- `sub_dir_b/`：

## 关键文档
- `xxx_overview.md`：
- `xxx_flow.md`：

## 维护约定
- 新增子目录时必须补充本文件
- 新增模块文档时需说明职责、流程、依赖
```

### B. 功能说明文档模板

```md
# <模块名> 说明

## 职责

## 核心流程
1.
2.
3.

## 关键依赖
-

## 数据结构/配置点
-

## 相关代码
- `<相对路径>`
```

## 输出要求

执行结束时提供：

1. 已创建目录树（简洁列表）
2. 每个目录对应的 `data_structure.md` 路径
3. 已生成的功能说明文档列表
4. 尚未创建但建议创建的子模块目录（若用户暂未确认）
5. 若是单模块模式：明确列出“模块范围确认依据”（用户提供入口或目录/文件名推断结果）
6. 已选初始化模式（轻量/全量）及执行边界
7. token 成本提示（尤其全量模式）
8. 若为轻量模式：后续“逐步构建”建议顺序（优先模块与建议文档）

## 失败与降级处理

- 若无法确认项目根目录：先向用户确认路径，不盲目创建。
- 若用户要求单模块创建但未给入口目录：先做目录名/文件名级别推断并请求确认，确认前禁止读代码。
- 若目录名/文件名也无法推断出候选入口：明确阻断并给出最小补充信息清单（项目根路径、模块别名、任一入口目录）。
- 若读取代码文件失败：记录失败原因，并先完成可创建的索引骨架。
- 若用户只想创建骨架不写内容：允许仅创建目录 + `data_structure.md`，并明确哪些说明文档待补。
- 轻量模式下若无法从目录名/文件名获得足够证据：必须阻断并请求补充信息，禁止凭经验假设目录结构。
