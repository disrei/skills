# 项目知识库更新 Skill（project-kb-update）

## 目标

在 `knowledge/` 已存在的前提下，持续维护与更新项目知识库，确保：

- 更新内容符合现有知识库层级与命名结构
- 新知识来自“当前实现 + 当前会话确认信息”，并沉淀到合适文档
- 对偏离知识库范畴的更新请求先提示风险，再由用户确认是否继续

## 严格触发条件

仅当同时满足以下条件时调用本 Skill：

1. 用户明确表达“更新/维护/补充/同步 knowledge 知识库”或同义意图。
2. 项目根目录已存在 `knowledge/`。

若 `knowledge/` 不存在：

- 不执行更新流程。
- 明确告知需要先初始化知识库，并建议使用 `project-kb-init`。

## 执行原则

- 结构先行原则：更新前必须先读取 `knowledge/` 结构与各层 `data_structure.md`，再决定落盘位置。
- 证据驱动原则：仅沉淀有依据的知识（代码实现、配置、测试输出、用户明确说明）。
- 对话沉淀原则：从当前会话提炼稳定结论，写入对应知识文档，避免只留在聊天记录。
- 边界守护原则：若待更新内容偏离当前知识库范畴，必须先说明原因并询问是否继续写入当前知识库。
- 显式越界优先原则：当用户请求本身就出现明显跨域信号（例如在游戏知识库中写入计费/财务/HR/法务流程），应先做越界提示与确认，不要先承诺直接落盘。
- 决策变更一致性原则：若用户在后续对话中改变越界处理决定（从不写入改为写入，或反向），必须先回顾上一次决策与当前变更，再给新的执行计划，避免状态混乱。
- 并发安全原则：同一知识库文件在同一时段只允许一个更新流程落盘；后续并发请求先检查 pending 标记，再决定直接更新或写入 cache。
- 一致性原则：遵循现有命名与目录风格，不随意重命名核心目录。
- 最小改动原则：优先增量更新；只有必要时才新增目录或重组文档。

## 标准流程

### 1) 前置检查

1. 识别项目根目录（优先用户指定路径，否则当前工作目录）。
2. 检查 `<project_root>/knowledge` 是否存在。
3. 若不存在：阻断更新并引导初始化。

额外约束：

- `knowledge/` 不存在时，禁止“自动先初始化再更新”。必须先让用户明确走 `project-kb-init`。

### 2) 读取知识库结构（必须）

更新前至少完成以下读取：

- `knowledge/data_structure.md`
- 所有一级目录下的 `data_structure.md`
- 与本次更新目标最相关的现有说明文档（如 `*_overview.md`、`*_flow.md`）

读取后输出一段“结构理解摘要”，至少包含：

- 当前知识库的目录分层与职责边界
- 本次最可能落盘的目标目录
- 可能受影响的既有文档列表

### 2.1) 并发检查与 pending 标记（必须）

每次准备更新目标知识文件前，先执行并发流程（批次一致性模式）：

1. 先识别“本次更新批次”的全部目标文件。
2. 对每个目标文件定位其上层 `data_structure.md`（用户口语可能写成 `data_struct.md`，统一按 `data_structure.md` 处理）。
3. 扫描这些 `data_structure.md` 的 pending 标记。
4. 若任一目标文件命中 pending：本批次进入 `batch_cache_mode`，本次所有目标文件都不直接落盘，统一写入 cache。
5. 仅当本批次所有目标文件都未命中 pending 时，才允许写入 pending 标记并继续正式更新流程。

pending 标记最少记录：

- 目标文件路径
- 更新会话标识（或任务标识）
- 标记时间
- 状态（pending / done）

stale 判定建议：

- 若 pending 持续超过 24 小时且无完成记录，视为 stale pending。
- 命中 stale pending 时，不得直接覆盖；先提示用户选择“接管并清理旧 pending”或“继续仅写 cache”。

stale 提示模板（可直接使用）：

`检测到旧的 pending 标记（超过 24 小时未完成）。你可以选择：A. 接管并清理旧 pending 后继续更新；B. 暂不接管，本次仅写入 cache。`

提示模板（可直接使用）：

`检测到本批次中至少一个目标文件正在被更新。为保证版本一致性，本次涉及的所有目标文件将统一写入本地 cache；下次更新时我会先合并 cache 再统一更新。`

### 2.2) cache 文件机制（并发降级）

当进入 `batch_cache_mode` 时：

1. 将本批次全部目标文件的更新请求写入本地 cache 文件（建议路径：`knowledge/.cache/pending_updates/`）。
2. cache 文件命名建议：`<timestamp>_<target-file-slug>.json`。
3. 下次进入更新流程时，先扫描该目录并加载相关 cache 项，再与新请求合并处理。

批次一致性要求：

- 进入 `batch_cache_mode` 后，不允许“部分文件直接更新、部分文件写 cache”的混合执行。
- 只有在后续流程中确认该批次可统一落盘时，才从 cache 迁移到正式更新。

cache 文件最少字段：

- `target_file_path`：目标知识库文件路径
- `knowledge_points`：待更新知识点清单
- `source`：来源（实现来源 / 对话来源）
- `created_at`：创建时间
- `status`：pending / merged / dropped
- `batch_id`：同一批次更新标识（用于保证整批一致处理）

### 2.3) cache 合并冲突处理（必须）

当同一 `target_file_path` 存在多条 cache 更新项且知识点冲突时：

1. 先输出“冲突清单”（冲突知识点、各来源、各版本结论）。
2. 冲突项默认不自动覆盖，先请求用户确认合并策略。
3. 非冲突项可先行合并，冲突项保持 pending。

合并策略选项（可直接给用户）：

- A. 以最新时间戳为准
- B. 以用户本轮明确结论为准
- C. 手工逐项确认

冲突提示模板（可直接使用）：

`检测到同目标文件的 cache 更新项存在冲突：<conflicts>。请确认合并策略：A 最新时间戳优先 / B 本轮用户结论优先 / C 逐项确认。确认前我只会合并非冲突项。`

### 2.4) 版本漂移处理（cache vs 主文件）

当准备合并 cache 时，必须校验目标主文件是否发生版本漂移：

1. 比较 cache 记录的 `base_version`（或 `base_hash`）与当前主文件版本。
2. 若一致：可按正常流程合并。
3. 若不一致（主文件已被外部改写）：标记为 drift，先输出漂移风险与处理选项，不得直接覆盖。

漂移处理选项（可直接给用户）：

- A. 基于当前主文件重放 cache 变更（推荐）
- B. 放弃该 cache 项（标记 dropped）
- C. 手工确认后再合并

漂移提示模板（可直接使用）：

`检测到 cache 与主文件版本不一致（version drift）：cache 基线=<base_version>，当前主文件=<current_version>。请确认处理方式：A 重放到当前版本 / B 放弃该 cache / C 手工确认后合并。`

### 3) 汇总待更新知识来源（必须）

将本次更新拆成两类来源：

- **实现来源**：本次代码改动、配置变更、流程变化
- **对话来源**：用户在会话中新增的规则、约束、业务结论

输出“更新输入清单”，示例：

```md
## 更新输入清单
- 实现来源：
  - <变更点 A>
  - <变更点 B>
- 对话来源：
  - <规则 1>
  - <规则 2>
```

### 4) 结构映射与落盘计划

将每条待更新知识映射到现有知识库路径：

- 能映射：写明目标文件（更新已有 or 新建文档）
- 不能映射：标记为“潜在越界项”，进入下一步边界确认

建议输出格式：

```md
## 更新落盘计划
1. <知识点> -> knowledge/<path-a>/<file-a>.md（更新）
2. <知识点> -> knowledge/<path-b>/<file-b>.md（新建）

## 潜在越界项
- <知识点 X>：偏离原因 <why>
```

### 5) 越界确认（必须）

若存在“潜在越界项”，必须先提示并询问用户：

1. 说明为何偏离当前知识库范畴（模块边界不匹配、职责不一致、项目外内容等）。
2. 提供可选处理方式：
   - A. 继续写入当前知识库（并标注临时归档位置）
   - B. 不写入，先调整知识库范围/结构
3. 在用户确认前，不得落盘越界项。

显式越界处理：

- 若用户请求在语义上已明显跨域（即使尚未完成完整结构读取），也要先给出“潜在越界项 + 原因 + 是否继续”三段式确认。
- 明确引用当前知识库的边界依据（目录职责或 `data_structure.md` 的定位语句），不能只说“感觉不合适”。

询问模板（可直接使用）：

`我发现以下待更新知识与当前 knowledge 范畴不完全匹配：<items>。原因是：<reasons>。要继续写入当前知识库吗？若继续，我会先写入 <temporary_path> 并标记待重构。`

用户改主意时的处理模板（可直接使用）：

`你已将该越界项决策从“<old_decision>”改为“<new_decision>”。我会按新决策执行：<plan>。同时保留决策变更记录，避免后续冲突。`

### 5.1) 越界项决策变更处理（必须）

当用户对同一越界项改变决策时，按以下顺序执行：

1. 回顾上一次决策与当前新决策。
2. 输出新的落盘/不落盘计划。
3. 若从“临时归档”转“正式并入”，先给迁移风险与迁移步骤。
4. 在更新输出中记录“决策变更条目”（时间、原决策、新决策、原因）。

连续反转约束：

- 对同一越界项（按“主题 + 模块范围”识别）发生多次改主意时，以“最新一次明确用户决策”为唯一生效决策。
- 旧决策必须标记为 superseded（被后续决策覆盖），避免同时执行互斥动作。
- 若同一条消息内出现互斥指令（例如“不要写入”与“立刻并入”同时出现），先阻断执行并请求用户给出单一最终决策。

同主题归一化约束：

- 对跨轮对话中的同义改写（如“HR 绩效流程”/“人事考核流程”）要先做主题归一化，避免误判为新事项。
- 若判定为同一越界主题，必须先回顾该主题历史决策，再按“最新明确决策”执行。
- 若无法确定是否同一主题，先给出候选归一化结果并让用户确认后再落盘。

### 6) 执行更新

按确认后的计划执行增量更新：

- 更新已有文档：保留原有有效信息，补充新变化与版本上下文。
- 新建文档：命名清晰，内容完整，避免空模板。
- 若新增目录：必须补对应 `data_structure.md`，并回填父目录索引。

推荐文档内容最少包含：

- 变更背景（为什么更新）
- 当前结论（更新后的事实）
- 影响范围（模块/接口/配置）
- 关联实现证据（代码路径、配置项、对话确认）

### 7) 一致性回写

更新完成后，至少回写：

- 受影响目录的 `data_structure.md`（文档列表、职责边界）
- 根目录 `knowledge/data_structure.md`（若一级结构或关键文档变动）

## 输出要求

执行结束时提供：

1. 结构读取摘要（你如何理解当前知识库）
2. 更新输入清单（实现来源 + 对话来源）
3. 实际更新文件清单（新增/修改）
4. 每个文件的更新摘要（1-3 行）
5. 越界项处理结果（若有）
6. 尚未更新但建议后续补齐的知识点
7. 并发处理结果（是否命中 pending、是否写入 cache、cache 文件路径）
8. 批次处理模式（normal / batch_cache_mode）与批次标识（batch_id）

## 失败与降级处理

- 若无法读取 `knowledge/` 关键索引：先报告缺失文件并仅做最小安全更新。
- 若实现信息不完整：先沉淀已确认知识，并将不确定项标记为“待确认”。
- 若用户拒绝越界写入：不落盘越界项，仅输出建议目录调整方案。
- 若更新目标与现有结构冲突严重：先提交重构提案，待用户确认后再执行。
- 若 pending 标记长期未清理：先提示可能存在中断更新，要求用户确认是否接管并清理旧 pending。
- 若检测到 cache 文件：先汇总 cache 与本次请求的合并计划，再执行落盘。
- 若本批次任一目标文件命中 pending：整批转 cache，不得拆分执行。
